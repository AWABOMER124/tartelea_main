# تقرير فحص مشاكل التسجيل وتسجيل الدخول (Tartelea Backend)

## ملخص تنفيذي
- سبب التعطل الأكثر تأثيرًا حاليًا هو فشل الاتصال بقاعدة البيانات (`ENOTFOUND`) بسبب `DB_HOST` غير قابل للوصول من البيئة الحالية.
- يوجد خلل إعدادات CORS كان ممكن يمنع طلبات المصادقة من المتصفح عند تفعيل `credentials`.
- هناك ثغرات جودة في مسار المصادقة (غياب تحقق على `/verify-email`) وتباين بين ملفات الـ schema قد يسبب أخطاء تشغيلية حسب بيئة النشر.

## الأخطاء والمشاكل المكتشفة

### 1) فشل الاتصال بقاعدة البيانات
- السجلات تحتوي تكرارًا لـ:
  - `Failed to connect to the database. Retrying in 5 seconds...`
  - `getaddrinfo ENOTFOUND tartelea-va-jjs8rs`
- هذا يمنع أي عملية `signup/login` لأنها تعتمد كليًا على DB.

### 2) إعداد CORS غير متوافق مع `credentials`
- كان الإعداد يستخدم `origin: '*'` مع `credentials: true`.
- في المتصفحات، هذا نمط غير صالح عندما ترسل الكوكيز/Authorization بطريقة تتطلب credentials.

### 3) متغيرات SMTP كانت غير معرفة في طبقة env
- خدمة الإيميل تعتمد `EMAIL_HOST/EMAIL_PORT/EMAIL_USER/EMAIL_PASS`.
- طبقة التحقق للبيئة لم تكن تضم هذه المفاتيح، فكانت الخدمة تعمل عمليًا في Mock mode بدون إرسال فعلي.

### 4) غياب التحقق من مدخلات `verify-email`
- المسار `/auth/verify-email` كان بدون `validate(...)`.
- يزيد احتمالية الأخطاء وسلوك غير متوقع عند مدخلات ناقصة/غير صحيحة.

### 5) ازدواجية/تضارب مخطط قاعدة البيانات
- يوجد ملفان: `schema.sql` و `src/db/schema.sql` مع اختلافات جوهرية (أعمدة وقيود وأنواع بيانات).
- هذا يسبب اختلاف سلوك بين البيئات بحسب أي ملف تم تطبيقه.

## ما تم إصلاحه مباشرة في هذا التحديث
1. دعم `DATABASE_URL` و`DATABASE_SSL` في الاتصال بقاعدة البيانات (لبيئات Dokploy/Heroku).
2. إصلاح CORS ليعتمد `ALLOWED_ORIGINS` بشكل صحيح بدل `*` مع credentials.
3. إضافة متغيرات SMTP إلى طبقة env حتى تعمل خدمة الإيميل كما هو متوقع.
4. إضافة schema تحقق لمسار `verify-email`.

## رودماب المعالجة والتحسين (مقترح عملي)

### المرحلة 0 (اليوم نفسه – إطفاء الحريق)
1. **تثبيت اتصال DB**
   - إن كانت البيئة تستعمل خدمة Managed DB: اضبط `DATABASE_URL` الصحيح.
   - إن كانت Docker Compose: اجعل `DB_HOST=db` داخل شبكة compose.
2. **التأكد من سريان CORS**
   - اضبط `ALLOWED_ORIGINS` إلى دومينات الويب الحقيقية (مفصولة بفواصل).
3. **اختبار smoke auth**
   - Signup → Verify Email → Login → Me.

### المرحلة 1 (1-2 يوم)
1. **توحيد schema**
   - اعتماد مصدر واحد للهجرة (Migration strategy) وإلغاء الازدواجية.
2. **تحسين أخطاء المصادقة**
   - رسائل خطأ معيارية (`AUTH_*` codes).
   - حظر Login قبل التفعيل إن كان ذلك شرط المنتج.
3. **اختبارات أساسية**
   - Integration tests لمسارات auth.

### المرحلة 2 (3-5 أيام)
1. **Observability**
   - Correlation IDs لكل request.
   - Dashboards لأخطاء auth/DB/email.
2. **أمان**
   - تشديد rate-limit لمسارات auth.
   - lockout/backoff لمحاولات login الفاشلة.
3. **جودة نشر**
   - CI checks: lint + tests + migration check.

### المرحلة 3 (تحسين مستمر)
1. **تحسين تجربة المستخدم**
   - إعادة إرسال كود التفعيل مع throttle.
   - رسائل حالة دقيقة في التطبيق.
2. **توثيق تشغيلي**
   - Runbook لحالات: DB down / SMTP down / JWT issues.

## مؤشرات نجاح (KPIs)
- نجاح `signup` > 98%.
- نجاح `login` > 99%.
- أخطاء DB connectivity في auth = 0.
- زمن استجابة auth p95 < 300ms (بدون عمليات خارجية بطيئة).
