# تقرير فحص شامل لتطبيق الهاتف + خارطة طريق تحسين احترافية

> تاريخ المراجعة: 2026-04-07
> 
> الهدف: تحديد القصور والمشاكل في التطبيق (التصميم، الأزرار، الصفحات، التقنية)، ثم وضع Roadmap إصلاح وتحسين **بدون تغيير هيكل ومحتوى التطبيق**.

---

## 1) ملخص تنفيذي

التطبيق يملك أساسًا بصريًا جيدًا وهوية عربية واضحة، لكن توجد نقاط ضعف حرجة تمنع الاعتماد الإنتاجي الكامل:

1. **مخاطر أمنية عالية**: مفاتيح حساسة مضمّنة داخل الكود، واستخدام API عبر HTTP غير مشفّر.
2. **جاهزية بناء Android غير مكتملة بالكامل في هذه البيئة**: بيئة الفحص الحالية لا تحتوي Flutter SDK، ومع ذلك تم تحسين تهيئة Gradle لتكون أكثر قابلية للنقل بين الأنظمة.
3. **ثغرات في تجربة الاستخدام**: شاشة البداية لا تعتمد حالة تسجيل الدخول، التحقق من المدخلات في شاشة المصادقة محدود، ورسائل الأخطاء تقنية وغير موجهة للمستخدم.
4. **اعتماد كبير على Mock/Fallback** في أجزاء مهمة (غرف صوتية/مصادقة/محتوى)، مما قد يسبب فرقًا بين سلوك بيئة التطوير وسلوك الإنتاج.
5. **دين تقني UI**: استخدام واسع لأنماط ألوان قديمة (`withAlpha`) وتكرار كتل تصميم كثيرة بدل توحيد مكونات التصميم.

---

## 2) نتائج الفحص التفصيلية

### A) الأمن والحماية

- مفاتيح Supabase وLiveKit موجودة داخل الملفات المصدرية مباشرة.
- يوجد `apiSecret` خاص بـ LiveKit داخل التطبيق العميل (يجب ألا يكون في تطبيق الجوال نهائيًا).
- عنوان API يستخدم `http://` بدل `https://`.

**التأثير:** خطر تسريب مفاتيح، إساءة استخدام الخدمات، واعتراض الطلبات الشبكية.

### B) البنية والتكامل

- بعض المستودعات تستخدم نمط Fallback إلى بيانات Mock عند الفشل الشبكي، وهذا جيد مؤقتًا لكنه يخفي أعطال الإنتاج.
- وجود أكثر من مسار/تنفيذ لمجالات متشابهة (مثل audio rooms/features + presentation) يحتاج توحيد تدريجي لتقليل التعقيد.

### C) جودة UX للصفحات والأزرار

- شاشة Splash تنتقل للصفحة الرئيسية بعد تأخير ثابت بدل الاعتماد على حالة الجلسة.
- شاشة Auth تستخدم `TextField` مباشر دون نموذج تحقق قوي (صيغة بريد، طول كلمة مرور، تأكيد كلمة مرور...).
- بعض تدفقات التنقل والرسائل فيها سلوك تقني للمستخدم النهائي (مثل عرض `e.toString()`).

### D) جاهزية Android

- تم حذف إعداد `org.gradle.java.home` المربوط بمسار Windows محلي، لأنه يسبب فشلًا في البيئات الأخرى (Linux CI / Mac).
- لا يمكن بناء APK داخل هذه البيئة بسبب غياب Flutter SDK (قيد بيئي وليس خطأ في الكود).

## 4) Roadmap احترافية (بدون كسر الهيكل الحالي)

## المرحلة 0 — Hotfix (1-2 يوم)
- نقل جميع المفاتيح الحساسة إلى بيئة آمنة (`--dart-define` + Remote config/Secrets manager).
- إزالة أي Secret من كود العميل فورًا (خصوصًا LiveKit server secret).
- فرض HTTPS لكل نقاط API.
- إضافة فحص CI يمنع تسريب secrets (gitleaks/trufflehog).

**معيار النجاح:** لا يوجد Secret في المستودع + كل الاتصالات مشفرة.

## المرحلة 1 — استقرار وظيفي (3-5 أيام)
- ربط Splash بقرار توجيه يعتمد الجلسة الفعلية (مصادقة/عدم مصادقة).
- تحسين Auth validation + رسائل أخطاء مفهومة للمستخدم.
- استبدال الأخطاء النصية التقنية برسائل UX-friendly مع سجل داخلي للمطور.
- إنهاء TODOs الأساسية في المصادقة (reset password endpoint عند توفره).

**معيار النجاح:** انخفاض أخطاء تسجيل الدخول وزيادة إتمام التدفق من أول محاولة.

## المرحلة 2 — تحسين التصميم والاتساق (5-7 أيام)
- توحيد مكونات الأزرار/البطاقات/الحقول ضمن design tokens الحالية بدون تغيير المحتوى.
- تقليل التكرار عبر Widgets مشتركة (PrimaryButton, EmptyState, SectionHeader).
- مراجعة الوصولية: contrast، أحجام الخطوط، targets للأزرار، حالات loading/empty/error.

**معيار النجاح:** واجهات أكثر اتساقًا + تقليل تكرار الكود.

## المرحلة 3 — الأداء والاعتمادية (4-6 أيام)
- ضبط إستراتيجية cache واضحة (TTL + invalidation) مع مؤشر يوضح مصدر البيانات (Live/Offline).
- إضافة telemetry للأعطال (Crashlytics/Sentry).
- اختبار سيناريوهات الشبكة البطيئة/المنقطعة لجميع الصفحات الرئيسية.

**معيار النجاح:** زمن فتح أفضل وسلوك متوقع في وضع الشبكة الضعيفة.

## المرحلة 4 — جاهزية إطلاق Android (2-3 أيام)
- إعداد flavors (dev/staging/prod) بدون تغيير هيكل الشاشات.
- توليد AAB + APK مع توقيع صحيح وإعدادات shrink/obfuscation مناسبة.
- تنفيذ smoke test على أجهزة حقيقية (Android 10, 12, 14) + قائمة فحص قبل التسليم.

**معيار النجاح:** حزمة Android قابلة للتثبيت والاختبار الفعلي على الهاتف.

## 5) خطة اختبار QA عملية (للنسخة الكاملة)

### تدفقات أساسية يجب اختبارها
1. فتح التطبيق → Splash → توجيه صحيح حسب حالة الجلسة.
2. تسجيل دخول بالبريد وكلمة المرور + Google.
3. استعراض الصفحة الرئيسية والمكتبة والتصفيات.
4. فتح تفاصيل المحتوى (مرئي/صوتي/مقال).
5. غرف صوتية: دخول/خروج/تفاعل.
6. الملف الشخصي وتحديث البيانات الأساسية.

### حالات الحواف
- انقطاع الإنترنت أثناء تحميل الصفحة.
- انتهاء صلاحية التوكن.
- رابط صورة غير صالح.
- Backend timeout و 5xx.

## 6) تجهيز نسخة Android كاملة الميزات للاختبار على الهاتف

> في هذه البيئة لم أتمكن من توليد APK فعلي لأن Flutter SDK غير مثبت.

### خطوات توليد النسخة على جهاز التطوير (محليًا)

```bash
flutter --version
flutter pub get
flutter analyze
flutter test
flutter build apk --release --dart-define=API_BASE_URL=https://YOUR_API
# أو
flutter build appbundle --release --dart-define=API_BASE_URL=https://YOUR_API
```

### التثبيت على الهاتف

```bash
adb devices
adb install -r build/app/outputs/flutter-apk/app-release.apk
```

### التحقق بعد التثبيت
- تسجيل دخول ناجح.
- تشغيل المحتوى المرئي/الصوتي.
- عمل التنقل السفلي وكل الصفحات الرئيسية.
- عدم ظهور أخطاء Crash خلال 15 دقيقة اختبار يدوي متواصل.
